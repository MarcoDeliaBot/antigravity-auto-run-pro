Diagnosi e Fix: Antigravity Auto Run Pro non clicca i pulsanti
Problema osservato

Dallo screenshot: l'estensione mostra Auto: ON e GOD nella status bar, ma il pulsante "Run Alt+Enter" nel pannello dell'agente NON viene cliccato automaticamente.

Cause radice identificate (analisi del codice)
🔴 Bug #1 — Webview Guard troppo restrittivo (causa principale)

File: extension.js, righe 65–71

js
content_copy
if (!document.querySelector('.react-app-container') &&
    !document.querySelector('[class*="agent"]') &&
    !document.querySelector('[data-vscode-context]') &&
    !document.querySelector('.antigravity-agent-side-panel') &&
    !document.querySelector('[class*="antigravity"]')) {
    return 'not-agent-panel';
}

Il Webview Guard deve trovare almeno uno di questi selettori nel DOM del pannello agente di Antigravity. Se l'IDE ha aggiornato la sua struttura DOM (classe CSS cambiata, rimozione di attributi) il guard restituisce 'not-agent-panel' e nessun pulsante viene cercato.

Dall'immagine si vede che il pannello è attivo — ma il guard potrebbe non riconoscerlo. La versione di Antigravity IDE potrebbe essere cambiata da quando sono stati scritti questi selettori.

🔴 Bug #2 — run non matchato quando testo è "Run Alt+Enter"

File: extension.js, riga 129:

js
content_copy
if (n.startsWith(t + ' alt+')) return true;

Il bottone mostra "Run Alt+Enter" non "Run Alt+D". Il suffisso alt+ funziona correttamente perché 'run alt+enter'.startsWith('run alt+') è true — questo non è il problema primario.

🟡 Bug #3 — getDirectText restituisce stringa vuota per bottoni con icone

Per <button><svg/> Run Alt+Enter</button> la struttura è:

nodo svg (tipo ELEMENT, non TEXT_NODE)
nodo testo " Run Alt+Enter"

getDirectText raccoglie solo i TEXT_NODE figli diretti, quindi dovrebbe funzionare — a meno che il testo "Run" non sia in un <span> figlio, nel qual caso getDirectText(button) restituisce stringa vuota.

Dato che 'run' ha text.length <= 4, il codice usa SOLO directText (non fullText):

js
content_copy
var checkText = (text.length <= 4 && !isExpandTarget) ? directText : (directText || fullText.substring(0, 60));

Se il testo "Run" è in uno <span> interno al button, getDirectText(button) = "" → nessun match → il pulsante non viene cliccato.

🔴 Bug #4 — ACCEPT_COMMANDS obsoleti
js
content_copy
const ACCEPT_COMMANDS = [
    'antigravity.agent.acceptAgentStep',
    'antigravity.terminalCommand.accept',
    'antigravity.terminalCommand.run',
    'antigravity.command.accept',
];

Se Antigravity IDE ha aggiornato i nomi dei comandi, questi comandi VS Code falliscono silenziosamente (vengono ignorati perché il messaggio di errore contiene "not found").

Fix proposti
Fix #1 — Webview Guard: aggiungi selettori generici di fallback

Aggiungere selettori più generici che non dipendano da classi specifiche dell'IDE:

js
content_copy
!document.querySelector('[class*="panel"]') &&
!document.querySelector('[class*="chat"]') &&
!document.querySelector('[class*="conversation"]') &&
!document.querySelector('button') // Fallback: qualsiasi pulsante nel pannello

CAUTION

Il selettore button è troppo generico — si attiverebbe in OGNI webview VS Code (es. Extensions, Settings). Meglio usare un approccio diverso: invertire la logica del guard e skipparla se ci sono pulsanti con testo "Run"/"Accept" nel documento.

Approccio migliore: rimuovere il guard o renderlo molto più permissivo, affidarsi invece all'assenza di pulsanti ('no-permission-button') come segnale naturale.

Fix #2 — getDirectText fallback per testo in <span>

Per target === 'run' e simili, quando directText è vuoto, usare fullText come fallback se la lunghezza totale è contenuta (< 30 char):

js
content_copy
var checkText = (text.length <= 4 && !isExpandTarget)
    ? (directText || (fullText.length < 30 ? fullText : ''))  // <-- fallback aggiunto
    : (directText || fullText.substring(0, 60));
Fix #3 — Aggiornare ACCEPT_COMMANDS con i comandi correnti di Antigravity

Verificare quali comandi VS Code Antigravity espone nella versione attuale e aggiornarli.

Proposed Changes
extension.js
Webview Guard — più permissivo

Modificare il guard per passare se ci sono bottoni con testo "run"/"accept" nel documento, invece di bloccare basandosi su classi CSS.

getDirectText fallback

Aggiungere fallback a fullText per target corti (≤ 4 char) quando directText è vuoto e fullText è breve.

Versione → 1.8.3
Open Questions

IMPORTANT

Per diagnosticare con certezza quale guard sta fallendo, serve aprire il log dell'estensione: Ctrl+Shift+P → "AntiGravity Auto Run Pro: Open Log File"

Se vedi 'not-agent-panel' → è il Webview Guard (Fix #1) Se vedi 'no-permission-button' → è il text matching (Fix #2) Se non vedi nessuna riga CDP → il polling stesso è bloccato

Procedo con tutti e 3 i fix in modo conservativo (senza rischi di regressione).
